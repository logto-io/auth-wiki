## ¿Qué es una clave de API?

Las claves de API se utilizan para identificar y autorizar la aplicación o servicio que realiza la llamada. Son típicamente de larga duración y estáticas hasta ser rotadas y a menudo tienen un conjunto fijo de permisos. Se usan principalmente para comunicaciones de servidor a servidor o para acceder a datos públicos, estos tokens generalmente no representan a un usuario específico.

## ¿Cómo funciona una clave de API?

Una clave de API es una larga cadena de caracteres generada por el proveedor de la API y compartida con los usuarios autorizados. Esta clave debe incluirse en el encabezado de la solicitud cuando se accede a la API. Las claves de API son simples y efectivas para necesidades de seguridad básicas. Por ejemplo, servicios populares como Google Maps API y AWS proporcionan claves de API para controlar el acceso y monitorear el uso.

```java
curl -GET https://api.example.com/endpoint -H "Authorization: api-key YOUR_API_KEY"
```

Las claves de API no son tan efectivas como otras formas de autenticación de API (API authentication), como OAuth y JWT, pero aún juegan un papel importante al ayudar a los productores de API a monitorizar el uso, ya que es el método más sencillo y ampliamente utilizado para asegurar APIs.

## ¿Cuáles son sus pros y contras?

### Pros

- Simples de implementar: Las claves de API son fáciles de implementar y usar. Involucran adjuntar una clave al encabezado de la solicitud, siendo un método sencillo para que los desarrolladores y clientes comprendan y utilicen.
- Fáciles de monitorizar: Las claves de API son fáciles de monitorizar. Puedes rastrear el uso de cada clave y revocarlas si es necesario.
- Límite de tasa efectivo: Las claves de API son efectivas para el límite de tasa. Puedes establecer un límite en el número de solicitudes por clave para prevenir abusos.
- Adecuado para datos no sensibles: Las claves de API son adecuadas para datos no sensibles o APIs disponibles públicamente, donde los requisitos de seguridad son menores.

### Contras

- Seguridad limitada: Las claves de API no son lo suficientemente seguras para datos sensibles, especialmente para aplicaciones del lado del cliente. A menudo se utilizan en comunicaciones de máquina a máquina (machine-to-machine).
- No adecuado para la Autenticación de usuarios: Las claves de API están vinculadas a aplicaciones o sistemas, no a usuarios individuales, lo que dificulta identificar usuarios específicos o rastrear sus acciones.
- Sin expiración del token: Las claves de API son típicamente estáticas y no expiran. Si una clave es comprometida, podría ser mal utilizada indefinidamente a menos que se regenere manualmente.

## ¿Cuáles son los casos de uso para las claves de API?

- Comunicación de servicio a servicio: Las claves de API son adecuadas para escenarios donde las aplicaciones necesitan comunicarse con APIs directamente a través de CLIs. Ej. Llamar a las APIs de OpenAI.
- APIs públicas: Al exponer APIs al público, las claves de API proporcionan un método sencillo de control de acceso.
- Configuración simplificada: Para necesidades rápidas y simples de autenticación (authentication), especialmente en la fase de desarrollo. A diferencia de la autenticación de máquina a máquina (Machine-to-machine authentication), las claves de API no requieren registro previo del cliente, y tampoco necesitan intercambiarse por un token de acceso (access token). Simplemente pasas tu clave de API como un parámetro en tu solicitud y simplemente funciona.

En escenarios del mundo real, el propósito más común al construir productos es la integración de producto. Aquí hay un caso de uso típico:

## ¿Cuál es la diferencia entre Tokens de Acceso Personal (PAT) y Machine-to-Machine (M2M)?

Al hablar de claves de API, los tokens de acceso personal (Personal Access Tokens) y machine-to-machine también pueden mencionarse juntos, ya que todos pueden acceder programáticamente a recursos de API a través de comandos CLI, o establecer comunicación entre servicios de backend.

### Tokens de Acceso Personal (PAT)

Un token de acceso personal es también una cadena pero representa ***la identidad y permisos de un usuario específico***, se genera dinámicamente tras una autenticación (authentication) o inicio de sesión exitoso, y típicamente tiene una vida útil limitada pero puede ser renovado. Proporciona control de acceso detallado a datos y capacidades específicas del usuario y son comúnmente utilizados para herramientas CLI, scripts, o acceso personal a la API. La principal diferencia es que es más específico y se usa para acciones específicas del usuario.

### Machine-to-Machine (M2M)

La comunicación M2M es cuando dispositivos intercambian automáticamente datos sin intervención humana en un sentido más amplio.

En el contexto de OIDC (o OAuth 2.0), las aplicaciones M2M utilizan el Client Credentials Flow, como se define en el [protocolo OAuth 2.0 RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749), que soporta protocolos estándar similares. Generalmente involucra una aplicación cliente (una máquina o servicio) accediendo a recursos, ya sea por sí misma o en nombre de un usuario. Es ideal para situaciones donde solo clientes de confianza pueden acceder a servicios de backend.
