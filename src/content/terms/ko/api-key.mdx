## API 키란 무엇인가요?

API 키는 호출 애플리케이션 또는 서비스를 식별하고 권한을 부여하는 데 사용됩니다. API 키는 일반적으로 회전되기 전까지 장기적으로 유지되며 고정된 권한 집합을 갖습니다. 주로 서버 간 통신이나 공용 데이터에 액세스할 때 사용되며, 이러한 토큰은 일반적으로 특정 사용자를 대표하지 않습니다.

## API 키는 어떻게 작동하나요?

API 키는 API 제공자가 생성하여 인증된 사용자에게 공유하는 긴 문자열입니다. 이 키는 API에 액세스할 때 요청 헤더에 포함되어야 합니다. API 키는 기본적인 보안 요구 사항에 대해 쉽고 효과적입니다. 예를 들어, Google Maps API와 AWS 같은 인기 있는 서비스는 API 키를 통해 액세스를 제어하고 사용량을 모니터링합니다.

```java
curl -GET https://api.example.com/endpoint -H "Authorization: api-key YOUR_API_KEY"
```

API 키는 OAuth와 JWT 같은 다른 형태의 API 인증 (Authentication)만큼 효과적이지는 않지만, API 제작자가 사용량을 모니터링하는 데 중요한 역할을 합니다. 이는 API를 보호하는 가장 간단하고 널리 사용되는 방법이기 때문입니다.

## 장단점은 무엇인가요?

### 장점

- 구현이 간단함: API 키는 구현하고 사용하기 쉽습니다. 요청 헤더에 키를 첨부하는 것으로, 개발자와 클라이언트가 이해하고 사용하기 쉬운 방법입니다.
- 모니터링이 용이함: API 키는 모니터링하기 쉽습니다. 각 키의 사용량을 추적하고 필요시 키를 폐기할 수 있습니다.
- 효과적인 속도 제한: API 키는 속도 제한에 효과적입니다. 남용을 방지하기 위해 키당 요청 수에 제한을 설정할 수 있습니다.
- 비민감 데이터에 적합: API 키는 보안 요구 사항이 낮은 비민감 데이터나 공용 API에 적합합니다.

### 단점

- 제한된 보안: API 키는 특히 클라이언트 측 애플리케이션의 민감 데이터를 보호하기에 충분히 안전하지 않습니다. API 키는 종종เครื่องหมายในเครื่องมือสื่อสาร machine-to-machine 방식에 사용됩니다.
- 사용자 인증 (Authentication)에는 적합하지 않음: API 키는 애플리케이션 또는 시스템에 연결되어 있어 개별 사용자를 식별하거나 그들의 행동을 추적하기 어렵습니다.
- 토큰 만료 없음: API 키는 일반적으로 고정되어 만료되지 않습니다. 만약 키가 손상되면 수동으로 재생성하지 않는 한 무한정 악용될 수 있습니다.

## API 키의 사용 사례는 무엇인가요?

- 서비스 간 통신: API 키는 애플리케이션이 CLIs를 통해 API와 직접 통신해야 하는 시나리오에 적합합니다. 예: OpenAI API 호출.
- 공개 API: API를 대중에게 노출할 때, API 키는 간단한 액세스 제어 방법을 제공합니다.
- 간단한 설정: 특히 개발 단계에서 빠르고 간단한 인증 (Authentication) 요구 사항에 적합합니다. machine-to-machine 인증 (Authentication)과 달리, API 키는 사전 클라이언트 등록이 필요 없으며, 액세스 토큰으로 교환할 필요도 없습니다. 요청의 매개변수로 API 키를 전달하면 간단히 작동합니다.

실제 시나리오에서 제품을 개발할 때 가장 흔한 목적은 제품 통합입니다. 여기에 일반적인 사용 사례가 있습니다:

### 예: Stripe와의 통합

Stripe는 다양한 플랫폼 및 애플리케이션과의 안전한 통합을 위해 API 키를 사용합니다. 이러한 키를 개발자 대시보드를 통해 생성, 조회, 삭제 및 관리할 수 있습니다. API 키를 사용하여 Stripe의 결제 및 청구 기능을 제품에 통합할 수 있습니다.

## Personal Access Tokens (PAT)와 Machine-to-Machine (M2M)의 차이점은 무엇인가요?

API 키에 대해 이야기할 때, 개인 액세스 토큰과 machine-to-machine도 함께 언급될 수 있습니다. 이들은 CLI 명령을 통해 API 리소스에 프로그래밍적으로 액세스하거나 백엔드 서비스 간의 통신을 설정할 수 있습니다.

### Personal Access Tokens (PATs)

개인 액세스 토큰도 문자열이지만 ***특정 사용자의*** 신원과 권한을 나타냅니다. 이는 성공적인 인증 (Authentication) 또는 로그인 시 동적으로 생성되며, 일반적으로 제한된 수명을 가지지만 갱신할 수 있습니다. 사용자의 특정 데이터 및 기능에 대한 세분화된 액세스 제어를 제공하며, CLI 도구, 스크립트 또는 개인 API 액세스에 일반적으로 사용됩니다. 주요 차이점은 사용자에 대한 특정 작업에 더 구체적으로 사용된다는 것입니다.

### Machine-to-Machine (M2M)

M2M 통신은 장치가 사람의 개입 없이 자동으로 데이터를 교환하는 것을 의미합니다.

OIDC (또는 OAuth 2.0) 문맥에서, M2M 애플리케이션은 [OAuth 2.0 RFC 6749 프로토콜](https://datatracker.ietf.org/doc/html/rfc6749)에서 정의된 Client Credentials Flow를 사용하며, 이는 유사한 표준 프로토콜을 지원합니다. 일반적으로 클라이언트 애플리케이션(기계 또는 서비스)이 독자적으로 또는 사용자를 대신하여 리소스에 액세스할 때 관련됩니다. 이는 신뢰할 수 있는 클라이언트만 백엔드 서비스에 액세스할 수 있는 상황에 이상적입니다.
